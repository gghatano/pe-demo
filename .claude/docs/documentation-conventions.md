# 実証研究レポートリポジトリの構成規約（汎用知見）

本ドキュメントは、情報系の**実証研究**（実験 + 評価レポート + 公開サイト）型リポジトリの
構成・運用知見を、研究テーマを問わず転用できる形で整理したものである。
`.claude/skills/` の各スキルはこの規約を前提に整理・レビュー・対応を行う。

> **本キット（academic-report-kit / パッケージ `ark`）でのレイアウト**:
> - **レポート md ソース**: `content/` 直下の `REPORT.md`（本体＝トップタブ）と補助タブ md
>   （`notation.md` / `method.md` / `related-work.md` / `extensions.md` / `engineering.md` / `faq.md`）。
> - **ビルダー / PAGES**: `ark/build_html.py`（実体）を `scripts/build_site.py` が呼ぶ。
>   「PAGES テーブル」は `scripts/build_site.py` の `pages`（単一の真実）。タブの追加・改名はここを更新。
> - **雛形**: `templates/*_TEMPLATE.md`。`bash scripts/init.sh` で `content/` に展開する。
> - **依存・環境**: `pyproject.toml` + `uv.lock`（uv で全依存をバージョン固定）。ビルドは
>   `uv run python scripts/build_site.py`（= `bash scripts/build.sh`）。
> - **実験コード**: テーマ依存の数値計算・図生成はキットに含めない。各リポジトリの
>   `experiments/` 等に置き、図は `content/figures/*.png` として本文から参照する。
> - `OWNER/REPO` 等のプレースホルダは対象リポジトリに合わせて差し替える。

---

## 1. ディレクトリ構成の原則

```
<repo>/
├── README.md            … 入口。スコープ宣言 + クイックスタート + 構成図 + 公開URL
├── content/             … 公開ページの Markdown ソース（唯一の編集対象）
│   ├── REPORT.md        … レポート本体＝トップタブ（論文調・章構成）
│   ├── notation.md      … 記法・用語（用語集・記号定義。本体からはインライン定義＋リンク）
│   ├── method.md        … 対象手法の詳細・アルゴリズム・直感
│   ├── related-work.md  … 先行研究・関連手法と本検証の位置づけ
│   ├── extensions.md    … 発展・追加実験（比較手法・他データ等への best-effort 拡張）
│   ├── engineering.md   … エンジニアリング情報（環境構築・再現手順・実装メモ。本体から分離）
│   ├── faq.md           … FAQ・メモ（よくある疑問・設計判断）
│   └── figures/         … 本文から参照する画像（実験コードが生成。HTML には base64 埋め込み）
├── htmls/               … ビルド成果物（直接編集しない。Pages 公開元）
├── ark/                 … 再利用コア（build_html: ビルダー / context: 作業ディレクトリ・config 解決）
├── scripts/             … build_site.py（PAGES でビルド）/ init.sh（雛形展開）/ build.sh
├── templates/           … 各タブの Markdown 雛形（*_TEMPLATE.md）
├── tests/               … pytest 単体テスト（ビルダーの不変条件）
├── pyproject.toml / uv.lock … 全推移的依存をバージョン固定（再現性の要）
├── docs/plans/          … 実験計画書（実験ごとに 1 ファイル。実施後は記録を追記して閉じる）
└── experiments/         … テーマ依存の実験コード（任意。生成物は再実行で再生成可能に）
```

要点:
- **ソースと成果物を分離する**: `content/`（手で書く）と `htmls/`（ビルドで生成）を混ぜない。
  成果物を直接編集した変更は次のビルドで消える。
- **タブ定義は一箇所に集約する**: ページ一覧・出力名・サブタイトル・ナビは
  `scripts/build_site.py` の `pages`（PAGES テーブル）が単一の真実。
- **環境は uv で固定する**: `uv sync` で誰でも同じ依存を再現できる状態を保つ。

## 2. 記述の分離原則（最重要）

レポートの信頼性は「どこまでが事実で、どこからが推定か」の区別で決まる。

| 区分 | 内容 | 置き場所 | マーカー |
|---|---|---|---|
| **前提** | 対象のドキュメント・論文・コードから読み取れる仕様 | REPORT.md 背景・設定章 | 出典タグ [n] を必ず付す |
| **結果** | 本実験での実測値 | REPORT.md 結果章 / extensions.md | 実行環境・シード・バージョンを明記 |
| **考察** | 実測からの分析・推定 | 考察章 / 各「所見」 | `> 🔎 **考察**` 等のブロックで明示 |
| **課題** | 残課題・適用範囲の限界 | 残課題章 + GitHub Issues | 残課題は Issue 化する |
| **エンジニアリング記録** | 動かすまでの問題と対応 | engineering.md | 「研究知見ではなく一次記録」と位置づけを宣言 |

- 研究的内容（前提・結果・考察）とエンジニアリング情報（環境構築・再現手順・実装問題）は
  **ページを分ける**。本体レポートにはリンクだけ残す。
- **トップページ（レポート本体）は検証対象を主軸にする**。レポート本体は
  「検証対象で何が分かったか」を主語にして書き、手法の**アルゴリズム固有の深掘りは
  個別ページ（`method.md`）に分離**する。本体には**要約＋個別ページへのリンク**だけを残す。
  - **本体に残す**: 検証対象に固有の発見・前提（例: ある条件が結果に効くという検証結果の要約、
    検証対象のパイプライン由来の挙動、適用範囲・コストと効果のトレードオフの確認）。
  - **個別ページへ移す**: 手法のアルゴリズム由来の性質（例: ある手法が別の手法を上回る理由＝
    その手法固有の設計、ある手法の構造的限界、計算コスト・パラメータ依存の詳細）。
  - 移設時は、本体側に「要約 1〜2 行＋該当ページへのリンク」を残し、個別ページの既存の節へ
    重複・矛盾なく統合する。出典タグ [n]・アンカー・図参照は壊さない。
- 単一シードの数値を断定的に書かない。揺れの定量化（マルチシード mean±std）を別実験として持ち、
  本文では「傾向を読む」よう注記する。
- スコープ外（評価していない条件・手法）は README 冒頭とレポート内の両方で宣言する。

## 3. 公開（ビルド & Pages）の原則

- `scripts/build_site.py` がビルダー `ark/build_html.py` を呼び、`content/*.md` → `htmls/*.html` を生成する。
  ページ一覧・出力名・サブタイトル・タブは**`scripts/build_site.py` の `pages`（PAGES テーブル）が単一の真実**。
  ページを追加・改名したら必ず PAGES を更新する（タブナビは PAGES から自動生成する）。
- 図は base64 埋め込みで自己完結 HTML にし、HTML 単体で共有可能にする。
- 公開は GitHub Actions（`.github/workflows/deploy-pages.yml`）で `htmls/` をデプロイ。
  Pages の build_type は **workflow**（gh-pages ブランチ公開ではない）。`gh api repos/OWNER/REPO/pages -X PUT -f build_type=workflow` で設定。
  重い処理（実験・図生成）は公開ビルドに含めず、Markdown 変換のみで数十秒に収める。
- Markdown 内の相互リンクは `content/` 内の md 名（`faq.md` 等）で書く。
  ビルダーが公開後の HTML 名（`faq.html`）に書き換える。

### 3.1 二層構成（クリーンな本体 + 補助タブ）

情報系の論文・レポートとしての見え方を保つため、サイトは **「論文体の本体（REPORT）＋補助タブ」** の
二層にする。本体は最後まで通読できる学術文書として汚さず、実務ノイズは補助ページへ退避し、
本体からは 1〜2 行の要約＋リンクで参照する。

- **本体 REPORT.md**: 要旨 → イントロ（貢献を箇条書き）→ 背景・対象手法の概要 →
  実験設定と評価方法 → 実験結果（事実／考察を分離）→ 考察 → 残課題 → 結論 → 参考文献、の順。
  用語・数式の導出・関連研究の詳細・環境・FAQ は本体に書かず、補助ページへ出す。
- **補助タブ（標準セット）**: `notation`（記法・用語）/ `method`（手法・背景）/ `related-work`
  （関連研究）/ `extensions`（発展・追加実験）/ `engineering`（環境・再現・落とし穴）/ `faq`（FAQ・メモ）。
- **逸脱・別アプローチ・拡張は補助タブに出して本体を守る**: 本体は確定した主設定（または論文準拠）に保ち、
  比較手法・他データ・別設定への best-effort 拡張や、主設定から外れるロバストネス確認は `extensions` に置く。
  本体からは要約＋リンクで触れる。反省・経緯の自己言及（「当初は誤って…」等）は基本書かない——
  採用した設定を事実として述べ、別案は補助タブで扱う。
- **UI 実現**: 全ページ共通で (a) ヒーロー帯（サイト名＋ページ別サブタイトル）、(b) その下端に
  重ねる**上部タブナビ**（現在ページに `active`）、(c) **目次サイドバー**（h2/h3 から自動生成、sticky）、
  (d) 本文は白いカード。CSS・MathJax は全ページに同一ブロックを差し込む。レスポンシブで 1 カラム化。

### 3.2 体裁の約束（出典チップ・注記・数式）

- **出典チップ**: 事実主張には Markdown で `[[n]](#refn)` と書く（→ `<a href="#refn">[n]</a>` に展開、
  `a[href^="#ref"]` で上付き小チップに整形）。各ページ末尾に参考文献節（本体は `## 参考文献`、補助タブは
  `## 参照`）を置き、raw HTML `<ol class="refs"><li id="refn">…</li></ol>` でアンカーを張る。チップ→参照へジャンプできる。
- **注記/補足**: blockquote の先頭に絵文字を付けて種別を示す（📘 出典・根拠 / 💡 直感・ヒント /
  ⚠️ 注意・落とし穴 / 🔎 考察 / 📑 凡例）。事実章と考察章は語調も変える（考察は「〜と考えられる」）。
- **数式**: MathJax v3 + `pymdownx.arithmatex`（`generic: True`）。インラインは `$...$`、表示は
  独立行に前後空行を空けて `$$...$$`。**表示数式をリスト項目のインデント下に置くとブロック認識されず
  `$` が残る**ため、リスト外の独立段落に出す。

## 4. 再現性の原則

- 依存は**推移的依存まで**ロックする（`uv lock` で `uv.lock` を生成・コミット）。git 依存はコミットハッシュ固定。
- 上流ライブラリへの修正が必要なら最小パッチに留め、適用手順（setup スクリプトでの自動適用等）と
  理由・影響を `engineering.md` に記録する。
- 外部データ・モデルの取得元は URL とともに消失リスクを認識し、チェックサム検証・ベンダリングを
  バックログに積む。
- 実行環境（OS・言語/ランタイム・ハードウェア制約）を README と REPORT 冒頭の両方に明記する。

### 乱数再現性の2層モデルと再現ターゲット（exact / trend）

「動いた環境で seed 固定 → 同じ図が出る」は常に成り立つわけではない。乱数・非決定性を**2層**に分けて記述する。

- **非決定層**（ハードウェア非決定性・並列実行の順序・浮動小数点の非結合性・外部サービス・
  暗号論的に安全な乱数等）= **seed 固定でも揺れる**。同一 seed でも run 間でビット一致しないことがある
  （仕様としてそうなっている場合も多い）。どの処理がこの層かを明記する。
- **決定層**（データ分割・前処理・モデル初期化・ブートストラップ等の通常処理）= **seed 固定で再現可能**。固定 seed を明記する。

この区別に対応して、各実験は**再現ターゲット**を宣言する（計画段階で。手順は `experiment-plan`）:

- **exact**: 数値の完全一致を狙う（決定層のみ・決定的処理）。
- **trend**: 集約値が想定近傍に来る／特定現象が再現される、を成功条件とする（非決定層を含む場合の正当な目標）。
  単一値の一致ではなく集約指標の安定や、現象（例: ある条件で性能が頭打ちになる）の再現で判定する。

レポートには非再現性を**正直に明記**する（§2 の正直さ原則と接続）。「数値は単一 run の代表値」と書く。
クリーンルーム検証も、非決定層を含む場合は「同じ図がビット一致」でなく「スクリプトが完走し、集約指標が想定レンジに入る」を合格条件にする。

## 5. 課題管理の原則

- 残課題は REPORT の残課題章（読者向けの限界の説明）と
  GitHub Issues（`https://github.com/OWNER/REPO/issues`、開発者向けの追跡）の二層で持つ。
- 新しい残課題は Issue として起票し、`タイトル / ラベル / 本文（背景・やること・完了基準）` を備える。
  対応済みは Issue を close し、レポート側の残課題章からは該当行を消すか「→ #NN で対応」と追記する。

## 6. 実験の計画・実行の原則

レポート作成と実験は次のライフサイクルで進める。**レポート骨子の検討が先**で、
実験は骨子を段階的に埋めるために行う。

```
report-skeleton（骨子: 章立て・主張・必要な実験の抽出）
  → experiment-plan（計画: 指標の出典・公平な比較条件・評価シナリオ → docs/plans/）
  → experiment-run（実行: uv 固定環境・シード規約・結果のレポート反映）
  → doc-cycle（整理・レビュー・公開）
```

この全体は `research-cycle` スキルが統括する（現在地を判定し、該当プロセスから順次実行）。

### 再現追試の原則（論文準拠を一次情報とする）

タスクが**論文の再現追試**のときは、実験設計の前に **論文の実験セクションを一次情報として読み**、
データ（規模・由来）・前処理・主要指標（定義）・baseline・主要パラメータ・照合する図表番号と報告値を
抽出して計画書に固定する（手順は `experiment-plan` §0）。

- **一次情報の優先順位**: **論文 ＞ 公式リポジトリ/コード ＞ example・tutorial**。三者が食い違うとき
  「何を再現するか」は論文が正。コードの example は論文と前処理・指標・データ規模が異なることがある。
- **指標は論文の主要指標に合わせる**（独自指標は併記に留め、論文指標を必ず再現）。
- **逸脱は計画段階で明示する**: 論文設定から外れる選択（前処理の「改善」・指標の変更を含む）は、計画書の「逸脱ログ」に
  理由つきで記録する。本体レポートは採用した設定を事実として述べ、別アプローチ・改善・拡張は補助タブ
  （`extensions`）で扱う（§3.1）。本文で反省・経緯を長々と述べない。
- 仕上げに**論文の報告値（図表番号）と自分の結果を並べた表**を作り、一致・乖離を考察する。
- **best-effort 拡張**: 公式実装に無い比較手法の自前実装や、他データ・他設定への拡張は best-effort で行ってよい。
  ただし「絶対値の厳密一致は保証しない」旨を明記し、優劣・桁・傾向で評価する。再現できなかった点は正直に注記する。
- **完成前の独立レビュー**: 公開前に `report-review`（構成・図表提示順・俯瞰・冗長などアカデミック体裁・査読観点）を
  独立パスとして必ず通す。起草エージェントの自己点検だけで「完成」としない。

### モデルティアの規則（プロバイダ非依存）

役割ごとに要求するモデルの**ティア**を定める。具体的なモデル名はこの規則に書かず、
各エージェント定義（`.claude/agents/*.md`）の `model:` 1 行に集約する
（モデルやプロバイダが変わってもそこだけ差し替えればよい）。

| 役割 | ティア | エージェント | 理由 |
|---|---|---|---|
| 計画（骨子・実験計画・調査・issue 起案） | **上位** | `planner` | 判断の誤りが下流全体に波及する |
| レビュー（指摘の洗い出し・整合点検） | **中位** | `reviewer` | チェックリスト駆動で判断範囲が限定される |
| 実行・実装（スクリプト・環境・実験・ビルド） | **下位** | `implementer` | 計画とテストで制約され、量が多い |

下位ティアの裏付けとして、実行・実装側には機械的な制約（計画書・issue の完了条件・TDD・
計画外の条件変更の禁止）を課す。制約で品質を担保できない作業はティアを上げるのではなく、
まず計画・完了条件の側を具体化する。

### 作業単位の規則（worktree + issue）

- **並行作業は git worktree で行う**。worktree ごとに作業ブランチを切り、
  名前に issue 番号を含める（例: `git worktree add ../<repo>-issue-12 -b issue-12-<slug>`）。
- **worktree には対応する issue が必ず存在する**。issue のない worktree を作らない。
- **issue の内容は計画時に定める**（`experiment-plan` / planner の担当）:
  目的・完了条件（テスト可能な形）・担当エージェント・参照する計画書（`docs/plans/`）を本文に書く。
  既存の GitHub Issue があれば相互参照する。
- **worktree に対応する issue はサブエージェント（implementer）が担当する**。
  issue の完了条件の範囲だけを行い、範囲外は差し戻す。
  コミットメッセージで issue 番号を参照し、完了したら worktree を畳む（`git worktree remove`）。

### 実装の規則（TDD）

- 実装を伴う issue は **TDD** で進める: 完了条件を `tests/`（pytest）の失敗するテストとして
  先に書き、テストが通るように実装し、通った状態でリファクタリングする。
- テストにできない作業（実験実行・ビルド等）は、機械的な検証方法
  （生成物の存在・exit code・件数チェック）を issue の完了条件に定めて代える。

エージェント間の受け渡しは成果物（`docs/plans/` の計画書・issue・指摘リスト）を介して行い、
口頭の文脈に依存させない。

- **骨子先行**: 要旨と結論は最後に書く。骨子の TODO 残数で進捗を測る。
- **計画書を残す**: 実験ごとに `docs/plans/<ID>-<slug>.md`（目的・仮説・指標・条件・完了基準）。
  実施後は実行記録を追記して閉じる。計画なしの場当たり実行をしない。
- **指標は既存研究準拠**: 各指標に出典を付け、独自指標には理由を明記する。
- **公平な比較**: 同一データ・同一前処理・同一条件・同一シード集合。
  ベースライン（最単純手法）を必ず含め、チューニングの扱いを全対象で揃える。
- **評価シナリオを反映する**: 想定ユースケースを 1 行で書き、
  そのユースケースで実際に問われる指標・失敗モードを測る実験を追加する。
- **再現性は uv で担保**: 依存は推移的依存まで固定し（`uv.lock`）、
  実行環境のメタ情報（バージョン・シード）を生成物と一緒に保存する。

## 7. レビュー・対応の進め方

`.claude/skills/doc-cycle` が以下を順次実行する（個別に呼んでもよい）:

1. `report-review` — 前提・結果・考察・課題の分離と整合のレビュー（査読視点）＋日本語の AI 臭の除去（`stop-ai-slop-jp` を併用）
2. `repro-engineering-review` — 再現手順・依存固定（uv）・engineering タブのレビュー
3. `related-info-review` — 参考文献・関連研究・上流情報・バックログの整理
4. `publish-check` — content↔htmls↔ナビ↔README の整合確認とビルド検証

各スキルは「指摘リスト（重要度付き）→ ユーザー確認不要な軽微修正は適用 → 残りは報告」の流れで動く。
