# 新規データへ Tab-PE を適用するためのチェックリストと標準タスクリスト

新しい表形式データに Tab-PE を適用するときの一次資料。CSV を読んで既定の
`TabularEmbedding + L2` を設定するだけでは足りない場面が多いので、着手前に
このチェックリストで適用可否と設計方針を確認する。

このガイドは新規適用手順の起点である。案件ごとにこのファイルを見ながら、
`docs/datasets/<dataset>/` に個別の記入結果を残す。レポート本文へ載せる場合も
内容を二重管理せず、このファイルへリンクする。

各項目には区分を付ける。

- **必須**: これを満たさずに本番データへ適用してはいけない。
- **推奨**: 満たさなくても動くが、満たさないと結果の解釈や再現が弱くなる。
- **経験者レビュー必須**: DP・データ理解の経験者が見ないと危険な判断。

チェックボックスだけ埋めても意味がない。各項目の「なぜ」と「失敗例」を読んでから判断する。

---

## 標準埋め込みで足りるか、独自実装が要るか

まずここを判断する。下に行くほど独自実装が要る。

1. 全列が「独立した連続数値」か「順序を持たない少数カテゴリ」だけで、列間の制約もない。
   → 公式 `TabularEmbedding + L2`（min-max 正規化と one-hot）でよい。**推奨: まず公式で回す。**
2. 順序カテゴリ・ゼロ集中数値・裾の長い数値・列間の重複や整合性・地域の近さ・複数選択などが
   ひとつでもある。
   → 公式のままでは意味を表現できない。独自埋め込みを設計する（§4・§6・§7）。
3. 1 個人が複数行を持つ（トランザクション・時系列・明細）。
   → **経験者レビュー必須。** 行単位 DP のまま適用すると保証が意図とずれる（§1）。

公式で足りるケースでも、独自実装に進む前に必ず公式をベースラインとして回し、
数値で比較する。「公式では表現できないはず」という思い込みだけで独自実装を入れない。

# 適用可否チェックリスト

## 1. 適用対象・データ構造

- [ ] **必須** 原則として 1 行が 1 つの privacy unit を表すマスター形式である。
- [ ] **必須** 1 行が何を表すかを文章で定義した。
- [ ] **必須** 追加・削除される 1 単位（DP の隣接関係）を定義した。
- [ ] **必須** 同じ個人・世帯・企業が複数行に登場しないことを確認した。
- [ ] **経験者レビュー必須** 複数行に登場する場合、行単位 DP でよいか、
      個人・世帯・企業単位 DP が必要かを判断した。
- [ ] **必須** トランザクション・イベントログ・時系列・明細形式ではない。
- [ ] **経験者レビュー必須** トランザクション形式の場合、安易に適用せず、集約方法・
      系列埋め込み・1 主体あたりの寄与上限・user-level DP を設計できる経験者のレビューを受けた。

> なぜ: Tab-PE の DP は「1 行の追加削除」を単位に会計する。1 人が 100 行持つデータに
> 行単位でそのまま適用すると、その人の情報は 100 行分守られず、保証が名目より弱くなる。
>
> 失敗例: 購買明細（1 人が多数行）を行単位 DP でそのまま合成し、「ε=1 で安全」と説明して
> しまう。実際には個人単位の保証になっていない。

記入欄:

```text
1行の意味:
privacy unit:
隣接データセットの定義:
同一privacy unitの最大行数:
マスター / トランザクション / 時系列 / その他:
```

## 2. 識別子・機微情報

- [ ] **必須** 顧客 ID・社員番号・会員番号・メールアドレス・電話番号など直接識別子を入力特徴から除外した。
- [ ] **必須** 氏名・住所全文・自由記述など再識別性の高い列を除外または適切に変換した。
- [ ] **必須** ID らしい数値を連続値として距離計算へ入れていない。
- [ ] **推奨** 準識別子の組合せによる再識別リスクを確認した。
- [ ] **推奨** 合成後に ID が必要な場合は、元 ID とリンクしない新しい surrogate ID を付与する方針にした。
- [ ] **必須** 元の顧客 ID を合成レコードへ再付与しないことを確認した。

> なぜ: ID を連続値として距離に入れると、意味のない「ID が近い＝似ている」を学習してしまう。
> 後処理で足してよいのは連番・UUID などの新規 surrogate ID であって、元 ID との対応付けではない。
>
> 失敗例: `customer_id` を数値列のまま距離へ入れ、ID の並び順に沿った偽の近さが生まれる。

## 3. データ schema と domain

各列について次の表を埋める。

| 列名 | 業務上の意味 | 型 | feature/label/除外 | 許容範囲・カテゴリ | 欠損の意味 | 未知値方針 | privacy 上の取得方法 |
|---|---|---|---|---|---|---|---|

- [ ] **必須** 全列の意味をデータ所有者へ確認した。
- [ ] **必須** 数値列の単位・妥当範囲・外れ値を確認した。
- [ ] **必須** カテゴリ列の全想定値を確認した。
- [ ] **推奨** 本番データにだけ存在する値を想定した。
- [ ] **必須** metadata の値域・カテゴリ集合を公開 schema から取得できるか確認した。
- [ ] **経験者レビュー必須** 秘密データから値域・カテゴリ集合を抽出する場合、漏えいと DP 処理の要否を判断した。
- [ ] **必須** 列の型推定だけに依存せず、型を明示した。

> なぜ: 値域やカテゴリ集合を「手元の秘密データから」決めると、それ自体が秘密データの
> 関数になり、DP 会計の外で情報が漏れる。範囲は公開仕様・既知の domain bounds から固定する。

## 4. 列単位の意味構造

各列を次のいずれかに分類する。

連続数値 / カウント / 金額 / ゼロ集中数値 / 裾の長い数値 / 二値 / 名義カテゴリ /
順序カテゴリ / 階層カテゴリ / 地域・空間 / 日付・時刻 / 周期値 / 複数選択カテゴリ /
自由記述 / ID・除外対象。

### 順序カテゴリ

- [ ] **必須** 順序を持つカテゴリを one-hot のまま扱ってよいか判断した。
- [ ] **推奨** 順序を数値または独自埋め込みへ変換する場合、対応表を定義した。
- [ ] **経験者レビュー必須** 想定外カテゴリの位置づけを定義した。

例（学歴）:

```text
想定: 中卒 < 高卒 < 短大卒 < 大卒 < 院卒
実データ: 高専卒が存在
```

`高専卒` を未知カテゴリとしてカテゴリ 0 へ落とすと、順序上あり得ない場所に置かれる。
データ所有者と相談し、順序上の位置・別グループ・明示的な `unknown` のどれにするか決める。

### 地域・空間

- [ ] **必須** 地域を単なる one-hot で扱うか、階層・隣接・距離を反映するか判断した。
- [ ] **推奨** 都道府県・市区町村・地方などの階層を定義した。
- [ ] **経験者レビュー必須** 緯度経度・隣接関係・移動距離を使う場合、公開マスターと計算方法を固定した。

> なぜ: one-hot では東京・神奈川・大阪がすべて等距離になる。実際は東京と神奈川を、
> 神奈川と大阪より近く扱いたいことが多い。この「近さ」は one-hot では表現できない。

### 数値

- [ ] **必須** min-max・標準化・log・log1p・二値＋量・周期表現などの候補を比較した。
- [ ] **必須** 数値範囲が公開情報か、秘密データ由来かを記録した。
- [ ] **必須** 外れ値・範囲外値の clip 方針を定義した。
- [ ] **必須** 単位変換を固定した。

> なぜ: `capital-gain` のようにゼロが多く正値の裾が長い列を単純 min-max すると、
> 小さな正値の差がほぼ 0 に潰れ、距離に効かなくなる。「ゼロか否か」＋「正値の log 量」の
> ように分解すると差が残る（→ Adult の例は #24）。

## 5. 欠損・未知値・異常値

全列について次を明示する。

```text
欠損:       エラー / 専用カテゴリ / 補完 / 除外 / その他
未知カテゴリ: エラー / unknown / 公開マスターへ追加 / その他
数値範囲外:   エラー / clip / 拡張 / その他
不正な列間組合せ: エラー / penalty / 修復 / 許容
```

- [ ] **必須** 未知カテゴリを暗黙に先頭カテゴリへ写像しない。
- [ ] **必須** NaN/Inf を距離計算へ渡さない。
- [ ] **推奨** train には無く運用時に現れる値のテストを用意した。
- [ ] **推奨** 「欠損そのものに意味がある」場合を単純補完で消していない。

> 失敗例: `LabelEncoder` に未知値が来て暗黙に 0 番へ写り、「不明」が特定の実カテゴリに
> すり替わる。NaN がそのまま距離に入り、最近傍計算が壊れる。

## 6. 列間の関係・整合性

- [ ] **必須** 同じ情報を重複表現する列を特定した。
- [ ] **推奨** 条件付きで値が決まる列を特定した。
- [ ] **推奨** 許されない列の組合せを定義した。
- [ ] **必須** 距離上の重複計上を避ける方針を決めた。
- [ ] **推奨** 不整合を埋め込み penalty・後処理・生成 operator のどこで扱うか決めた。

例:

- `education` と `education-num`（同じ学歴を二重に表す）
- 郵便番号と都道府県
- 生年月日と年齢
- 商品カテゴリと商品コード
- 雇用形態と勤務時間

> なぜ: `education` の one-hot と `education-num` を両方入れると、学歴が距離を二重に支配する。
> さらに合成レコードで両者が食い違う（大卒なのに education-num が高卒相当）不整合が起きる。

## 7. 距離の統合・列間重み

- [ ] **必須** 列ごとの距離または埋め込み寄与を数式で記述した。
- [ ] **必須** 列単位の距離をどう統合するか定義した。
- [ ] **必須** 列数・カテゴリ数が多い属性だけが距離を支配しないことを確認した。
- [ ] **必須** 数値列とカテゴリ列の相対重みを定義した。
- [ ] **推奨** 重みの根拠を、業務知識・公開仕様・事前定義した仮説として記録した。
- [ ] **経験者レビュー必須** 秘密データ上の実験結果を見ながら無制限に重みを調整しない。

一般形:

\[
d(x,y)=\sqrt{\sum_j w_j\, d_j(x_j,y_j)^2 + \sum_k v_k\, c_k(x,y)^2}
\]

- \(d_j\): 列単位の距離
- \(w_j\): 列の重み
- \(c_k\): 列間不整合 penalty
- \(v_k\): penalty の重み

> なぜ: カテゴリ数の多い列（例: `native-country`）を素の one-hot にすると、その列だけで
> 次元数が増え、距離を支配してしまう。列別の重みで釣り合いを取る。
>
> 経験者レビューの理由: 秘密データの結果を見ながら重みを何度も回すと、その適応的選択自体が
> privacy を消費し得る。事前に候補を固定して比較する。

## 8. DP・実験設計

- [ ] **必須** metadata・値域・カテゴリ・重みの決定に秘密データを使ったか記録した。
- [ ] **経験者レビュー必須** 秘密データを使った前処理・探索が privacy accounting の外に漏れていないか確認した。
- [ ] **経験者レビュー必須** 同じ秘密データで複数設定を試す場合、適応的な選択による privacy 消費を検討した。
- [ ] **必須** privacy unit と vote normalization level が一致している。
- [ ] **必須** epsilon・delta・反復数・合成件数を事前に決めた。
- [ ] **必須** 公式条件と独自条件を別 experiment ID で保存する。
- [ ] **必須** 実際の非公開データを使う前に、公開または人工データで動作確認した。

> DP は「距離を変えても常に安全」ではない。距離・重み・metadata を秘密データから決めた瞬間、
> その決定も保護対象になる。この判断は経験者が見る。

## 9. 評価・公開判定

- [ ] **必須** utility・fidelity・意味的一貫性の指標を事前定義した。
- [ ] **推奨** 実データ train → 実データ test の Upper Bound を用意した。
- [ ] **必須** 合成データ train → 実データ test の評価を用意した。
- [ ] **必須** 複数 seed で平均・標準偏差を計算した。
- [ ] **推奨** 列ごとの分布だけでなく、重要な列間関係を評価した。
- [ ] **推奨** 禁止組合せ・不整合率を評価した。
- [ ] **推奨** 小さいセルや稀少カテゴリの出力を確認した。
- [ ] **経験者レビュー必須** DP 保証だけで自動的に公開可とせず、組織のデータ公開審査を実施した。

> なぜ: 非決定的な生成なので、単一 seed の 1 値を「達成」と言わない。複数 seed の mean±std で見る
> （→ #22）。accuracy が高くても少数クラスの macro F1 が低いことがある（本プロジェクトの
> Adult・Person Activity がその例）。

# 標準タスクリスト（Phase 0〜7）

新規適用は次の順で進める。各 Phase の成果物を残す。

## Phase 0: 適用可否判定

1. データ所有者と 1 行の意味・privacy unit・利用目的を確認する。
2. マスター形式か判定する。
3. トランザクション・時系列の場合は通常フローを止め、経験者レビューへ回す。
4. 実データを持ち出さずに schema・公開マスター・サンプルで検討できるか確認する。
5. 適用可否と保留条件を記録する。

成果物: `docs/datasets/<dataset>/applicability.md`

## Phase 1: データプロファイル

1. 列一覧・型・意味・単位・feature/label/除外を整理する。
2. ID・直接識別子・自由記述を除外する。
3. 値域・カテゴリ集合・欠損・未知値・外れ値を整理する。
4. 列間制約と重複表現を整理する。
5. 使う metadata の出典・version・hash を固定する。

成果物: `docs/datasets/<dataset>/profile.md`、`configs/datasets/<dataset>.yaml`

## Phase 2: 前処理

1. チェックリストを満たす前処理仕様を作る。
2. ID 除外・型変換・カテゴリ正規化・欠損処理・範囲外処理を実装する。
3. 順序カテゴリ・地域・日付・ゼロ集中数値を変換する。
4. 元データを上書きせず、入力・処理済み・出力を分離する。
5. 行数・欠損数・カテゴリ数・除外件数を監査ログへ残す。

成果物: `scripts/datasets/<dataset>/preprocess.py`、
`tests/datasets/test_<dataset>_preprocess.py`、前処理 manifest

## Phase 3: 距離・埋め込み設計

1. 公式 `TabularEmbedding + L2` をベースラインとして定義する。
2. 列ごとの表現と距離を設計する。
3. 欠損・未知値・範囲外値の挙動を定義する。
4. 列間制約・不整合 penalty を設計する。
5. 列間重みと根拠を定義する。
6. 埋め込み次元数と各列の寄与上限を計算する。
7. 小規模 fixture で「近いはずのレコードが実際に近いか」をレビューする。

成果物: `docs/datasets/<dataset>/distance-design.md`、
`pe_demo/embedding/<dataset>.py`、`tests/datasets/test_<dataset>_embedding.py`

## Phase 4: 実験実装

1. ベースラインと独自 embedding を切り替えられる runner を実装する。
2. seed・epsilon・delta・反復数・合成件数を CLI/config 化する。
3. 実験条件・commit SHA・データ hash・embedding config を summary へ保存する。
4. 元ログ・checkpoint・合成 CSV・manifest を保存する。
5. 失敗時も `FAILED` レコードを残す。

成果物: `scripts/experiments/run_<dataset>.py`、
`configs/experiments/<dataset>.yaml`、`results/summaries/<dataset>_runs.json`

## Phase 5: 検証実験

1. 公開・人工データまたは承認済み環境で smoke test する。
2. ベースラインと独自距離を同一条件で比較する。
3. 各条件を 3 seed 以上実行する。
4. utility・fidelity・整合性・runtime を集計する。
5. 想定外値・欠損・禁止組合せを確認する。
6. 仮説を支持・不支持・未判定で結論付ける。

## Phase 6: DP・成果物レビュー

1. privacy unit・隣接関係・vote normalization を再確認する。
2. 前処理・metadata・重み調整に秘密データを使った箇所を棚卸しする。
3. privacy accounting の対象外アクセスがないか確認する。
4. サマリ値を元ログ・artifact へ追跡できることを確認する（→ #19）。
5. 秘密情報・元 ID・token・ローカルパスが成果物へ混入していないことを確認する。
6. 公開可否を組織ルールに従って承認する。

## Phase 7: ドキュメント化

1. データ概要・前処理・距離式・重み・DP 条件を記載する。
2. ベースラインとの差分を記載する。
3. 実行コマンドと再現方法を記載する。
4. 未確認事項・制約・適用できない用途を記載する。
5. 実装 Issue・PR・結果ファイルを相互リンクする。

# 記入例: Adult

本プロジェクトの Adult デモを例に、要点だけ記入する（詳細な追加実験は #24）。

- **1 行の意味 / privacy unit**: 1 人の国勢調査回答者。隣接 = 1 人の追加削除。マスター形式。
- **除外列**: 直接識別子はデータセットに無い。`fnlwgt` は標本重みで人物の意味的類似性とは
  別物なので、距離での寄与を下げる候補（#24 H3）。
- **順序カテゴリ**: `education`（中卒〜院卒の順序）。`education-num` が既に順序表現なので、
  `education` の one-hot と二重計上になる（§6）。実データに `高専卒` 相当が来たら、
  カテゴリ 0 へ落とさず順序上の位置か `unknown` を明示する。
- **ゼロ集中・裾の長い数値**: `capital-gain` / `capital-loss`。単純 min-max では小さな正値差が
  消えるので「ゼロか否か」＋「正値の log 量」に分解する（§4、#24 H1）。
- **列間整合性**: `education` と `education-num` の不一致に penalty（§6、#24 H2）。
- **地域の例**: Adult には強い地域構造は無いが、一般には東京・神奈川・大阪を one-hot で
  等距離にせず、近接を反映したい（§4 地域）。
- **評価**: accuracy だけで判断しない。Adult は不均衡二値で accuracy 80.94% に対し
  macro F1 70.78。合成 train → 実 test で複数 seed の mean±std を見る（§9、#22）。
